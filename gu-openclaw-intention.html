Section 01
OpenClaw.ai — The Real Product

OpenClaw is an open-source AI agent software project by Peter Steinberger (former PSPDFKit CEO, now at OpenAI). It is not an Opta internal concept — it is a real, widely-used product that Matthew runs on his own hardware.

OpenClaw is an autonomous AI agent that runs locally on your machines and connects to you via Telegram, Signal, Discord, and WhatsApp. It has a gateway server, AgentSkills (100+ tool plugins), a node mesh for cross-device communication, and its own scheduling system. It is model-agnostic — each bot can run any provider. Matthew runs 7 of them across 3 physical machines.

Section 03
How the Naming Got Confused

Opta's codebase had an internal mechanism for tracking which caller was responsible for a given LMX inference request. Somewhere in development, this mechanism was named after the OpenClaw product — even though they are completely separate concerns.

What was wrong

Stale Internal Branding

Opta's bridge identity system used names like OPENCLAW_IDENTITY_PATHS, openclaw-scope.ts, X-OpenClaw-Agent-ID, and openclawScope for what is purely an internal LMX attribution mechanism. These names implied a connection to the real OpenClaw product that doesn't exist at that layer.

What is correct

Two Distinct Things

The real OpenClaw integration (skill invocations, AgentSkills API) should keep the OpenClaw name because it literally connects to OpenClaw bots. The bridge identity mechanism (scope → deterministic ID → attribution header) is Opta-internal infrastructure and should use Opta-native naming.

Section 04
How the Bridge Identity System Works

When an external caller (like a Telegram bot or a device command) triggers an LMX inference request through Opta's daemon, the system needs to know who asked for it. The bridge identity system creates a deterministic, stable ID from a scope seed string.

The deriveBridgeAgentId(scope) function takes any scope seed string (like a Telegram chat ID, a bridge session ID, or an actor identifier) and produces a stable, reproducible ID using SHA-256 — sliced to 24 hex characters and prefixed.
// src/utils/bridge-scope.ts const BRIDGE_AGENT_PREFIX = 'opta-bridge'; const BRIDGE_AGENT_HASH_LENGTH = 24; export function deriveBridgeAgentId(scope: string): string { const digest = createHash('sha256') .update(scope.trim()) .digest('hex') .slice(0, BRIDGE_AGENT_HASH_LENGTH); return `${BRIDGE_AGENT_PREFIX}-${digest}`; }

Where the ID Gets Injected

When a bridge mutation targets an inference endpoint (/v1/chat/completions, /v1/completions, /v1/responses), the bridge worker injects the agent ID into the request automatically — unless the caller already provided their own identity.

// src/daemon/bridge-worker.ts — applyBridgeIdentityToMutation() const isInferencePath = BRIDGE_IDENTITY_PATHS.has(mutation.path); const callerHasIdentity = mutation.headers?.['x-client-id'] !== undefined; if (isInferencePath && !callerHasIdentity) { mutation.headers = { 'X-Opta-Bridge-ID': bridgeAgentId, // attribution 'X-Client-ID': bridgeAgentId, // LMX client tracking }; (mutation.body as Record)['user'] = bridgeAgentId; // OpenAI API 'user' field }
