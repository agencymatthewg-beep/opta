What it is

The feature

opta ceo-bench is an autonomous AI evaluation system built directly into the Opta CLI. It runs the active language model (via agentLoop) against a set of coding tasks in isolated temporary directories, then verifies the output with bash scripts.

The name "CEO" refers to a specific Opta autonomy mode — not the operator title. CEO mode sets: autonomy level 5 (maximum), headless continue, dangerous mode (auto-approve edits), executive report style, live data requirement, and policy audit enabled. It's designed to evaluate "full autonomy" model behaviour.

Intent vs reality

✓ What's working as intended

  • CEO autonomy mode correctly wires L5 config — requireLiveData, reportStyle: 'executive', policy.audit.enabled, headlessContinue
  • Ephemeral workspace per task with proper cleanup — mkdtemp + rm -rf in finally
  • OPIS context seeded into each workspace (APP.md, INDEX.md)
  • Loop control: CeoBenchTraceCollector detects identical tool-call sequences repeating ≥3 turns and aborts via AbortController
  • Trace artifacts persisted at docs/evidence/ceo-bench/ — full observability into model behaviour per task
  • JSON output mode for CI pipeline integration
  • --model override for cross-model comparison
  • Infrastructure unit tests: prompt generation, workspace seeding, loop detection, trace persistence, turn accounting

✗ What's missing or misaligned

  • Only 2 tasks — math-function (create a file with add()) and failing-test (fix a wrong operator). Both trivially simple, not CEO-grade
  • Tasks are hardcoded in DEFAULT_TASKS — no external task file support (JSON/YAML)
  • setupScript and verifyScript use execa('bash', …) — fails on Windows
  • No per-task timeout — a hung task blocks indefinitely (loop detection only catches repetitive tool calls, not slow-running processes)
  • No efficiency scoring — turns and duration logged but not weighted into a composite score
  • Tasks run sequentially — no parallelism, slow for large suites
  • No baseline tracking — can't compare a run against a previous run or a baseline model
  • Integration test for runCeoBenchmark() is in a separate runtime test file — unclear if it mocks the agent or runs live
  • benchmark/pages.ts (news/landing/chess benchmark) is a completely separate system with no connection to ceo-bench — confusing dual benchmark identity

Genuine strengths
Best-In-Class

Loop Detection System

The CeoBenchTraceCollector normalises tool call args (stable JSON key sort), builds turn signatures, and fires an AbortController when the same pattern repeats ≥3 times. Most benchmarks (SWE-bench, HumanEval) don't have this — they just let models spin.

Excellent

Ephemeral Workspace Isolation

Each task gets a fresh mkdtemp directory, seeded with OPIS docs, and cleaned in a finally block. CWD is restored to originalCwd. This prevents test pollution and matches how real CI evaluations work.

Excellent

Full Observability via Trace Artifacts

Every run produces a versioned JSON trace at docs/evidence/ceo-bench/ with per-turn tool calls, durations, errors, loop signals, and timestamps. This is better than most AI eval frameworks that only log pass/fail.

Excellent

CEO Autonomy Mode Integration

Uses the real computeAutonomyConfigUpdates system with mode: 'ceo' — executive reports, live data requirement, policy audit enabled, L5 autonomy. Benchmark actually runs in the same mode as real CEO-level tasks.

Good

Infrastructure Unit Tests

Prompt generation, workspace seeding, loop detection (including partial turn flush), trace persistence, and turn accounting are all unit-tested. The test file tests real behaviour — not just types.

Good

CI-Friendly Design

--json flag produces machine-readable output. --filter allows targeted runs. --model enables cross-model comparison. All designed for pipeline integration — though the pipeline isn't wired yet.

Gaps — by severity
Critical

Only 2 benchmark tasks

The benchmark has math-function (create a file) and failing-test (fix a wrong operator). These are sub-LeetCode-easy. A CEO benchmark should validate autonomous work across at minimum: multi-file refactoring, test writing, API integration, bug triage in unfamiliar code, documentation generation, security review, and performance analysis. Currently none of these are covered.

Missing: multi-file edit Missing: test writing Missing: bug triage Missing: refactoring Missing: API integration Missing: docs generation Missing: security review
Critical

bash-only scripts break Windows

Both setupScript and verifyScript run via execa('bash', ['-c', script]). Windows doesn't have bash by default. Every task in the benchmark is currently Windows-incompatible. Given the active Windows development track, this is a blocker.

Moderate

No per-task timeout

Loop detection catches repetitive tool calls but not a model that's making slow or different calls on each turn. A task that takes 40 slow turns blocks the entire benchmark. SWE-bench uses a 5-minute wall-clock timeout per task. A reasonable default here would be ~3–5 minutes.

Moderate

No baseline tracking or model comparison

Each run produces trace artifacts but they're not aggregated or compared. There's no way to answer "did upgrading from Kimi K2.5 to Claude Opus improve the pass rate?" without manually reading JSON files. A simple channels/ceo-bench/baseline.json pattern would solve this.

Moderate

Hardcoded task definitions — no external task files

DEFAULT_TASKS is a TypeScript array. Adding new tasks requires editing the runner source. Should support loading tasks from benchmark/ceo/tasks/*.json or similar, so the benchmark suite can grow without touching runner code.

Nice to Have

No efficiency scoring

Pass/fail is tracked but not turn efficiency or token usage. A model that passes in 2 turns is better than one that passes in 18. A composite score (pass × efficiency weight) would make ceo-bench a better model selection tool.

Nice to Have

Dual benchmark identity confusion

benchmark/pages.ts is a completely separate benchmark (news briefing, landing page, chess arena) that has nothing to do with ceo-bench. It uses a different runner, different output format, and different evaluation criteria. These should be clearly separated in the directory structure and documentation.

Optimisation roadmap — raise to 80+
1
Add 10–15 real benchmark tasks across diverse categories
Expand DEFAULT_TASKS (or external task files) to cover: multi-file refactor, test writing from scratch, fix a TypeScript type error, add a REST endpoint, debug an async race condition, write API docs from code, extract a reusable module. Each task should have a meaningful verify script. This alone would raise the Industry Parity score from 2.2 → 6+.
2
Cross-platform verification — replace bash with node or cross-env
Change the runner to detect OS and use node -e or execa('node', ...) for verification instead of execa('bash', ...). For tasks that must use shell, use cross-env or a shebang-aware runner. This unblocks Windows benchmarking immediately without changing the task interface.
3
Add per-task wall-clock timeout
Wrap the agentLoop call in a Promise.race against a configurable timeout (default 5 min). Pass the same AbortController signal to both. Log timeout as a distinct failure mode in the trace artifact — not just a generic error. This prevents benchmark hangs and adds a new efficiency signal.
4
External task file support
Load tasks from benchmark/ceo/tasks/ — one JSON file per task category. The runner should merge file-loaded tasks with any built-in defaults. This decouples task authoring from runner development and lets the task corpus grow without code changes or tests.
5
Local baseline tracking
After each run, write or update docs/evidence/ceo-bench/baseline.json with: run timestamp, model, pass rate, avg turns, avg duration. The report output should show delta vs previous baseline. This makes ceo-bench useful as a model upgrade/regression signal — its core purpose as a CEO evaluation tool.
6
Clarify dual benchmark identity in the codebase
benchmark/ceo/ (autonomous coding eval) and benchmark/pages.ts (news/landing/chess HTML generation) are completely separate systems with different purposes. Rename pages.ts to html-bench/ or similar and add a clear README in benchmark/ explaining the two tracks. Reduces confusion for anyone reading the code.
7
Efficiency score in results
Add a composite score to each CeoBenchResult: score = passed ? Math.max(0, 100 - (turns - 1) * 5) : 0. A 1-turn pass = 100, a 10-turn pass = 55, a 20-turn pass = 5. Report per-task scores and a suite average. This turns the benchmark from binary pass/fail into a model selection tool.
Final verdict
54
/ 100

The infrastructure is production-ready. The benchmark payload is not.

opta ceo-bench has the best loop-detection, workspace isolation, and trace observability of any custom benchmark I've seen — genuinely ahead of SWE-bench in those areas. The CEO autonomy mode integration is exactly right.

But a benchmark with 2 tasks cannot validate model capability. It's like having a Formula 1 car on a 20-metre track — the vehicle is exceptional but there's nowhere to go. The immediate priority is expanding the task corpus to 15–20 tasks across different categories, followed by cross-platform verification and a timeout mechanism.

With those changes, the score would realistically reach 75–80. With baseline tracking and efficiency scoring, 85+. The architecture is already there — it just needs content.
