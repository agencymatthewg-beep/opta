#01 Decompose the Big Four Files
~4 hours v0.9.1 HIGH
Four files hold 55% of app logic: MarkdownContent (1,514 lines), ChatViewModel (1,082), BotMapView (797), ChatView (726). Each mixes unrelated concerns — parsing with rendering, session logic with event handling, layout with data fetching. Split each into focused extensions.
Pros
  • Enables targeted unit testing per concern
  • Faster Xcode autocomplete and indexing
  • Parallel development without merge conflicts
  • Pure refactor — zero behavior change, zero risk
  • Makes future features easier to add
Cons
  • Large git diff (looks scary, but is mechanical)
  • Need to update Xcode project file references
  • Extensions can't add stored properties (design around)
✅ DO — v0.9.1. Pure refactor with massive maintainability payoff. No risk if done file-by-file with build verification between each.
MarkdownContent.swift → 3 files ChatViewModel.swift → 4 files BotMapView.swift → 3 files ChatView.swift → 2 files
#02 Complete VoiceOver Accessibility (WCAG AA)
~6 hours Blocks v1.0 CRITICAL
Only ~20% of interactive elements have VoiceOver labels. GUARDRAILS.md line 277 mandates all interactive elements must be labeled. Missing: .accessibilityHint() on ~100 buttons, .accessibilityValue() for dynamic states, element grouping for complex views. App Store could reject without this.
Pros
  • Unblocks App Store submission (Apple reviews accessibility)
  • Opens app to visually impaired users
  • Satisfies GUARDRAILS.md compliance requirement
  • Improves SwiftUI test automation (labels = test anchors)
  • Legal protection (ADA compliance)
Cons
  • Tedious, repetitive work across 23 view files
  • Hard to verify without real VoiceOver testing on device
  • May expose layout issues when Dynamic Type scales up
✅ DO — v0.9.1 / v1.0. Non-negotiable for App Store. Start with high-traffic views (ChatView, BotMapView, Settings), then complete remaining.
🛡 GUARDRAILS A04/A05 — All interactive elements require VoiceOver labels. Color contrast must meet WCAG AA 4.5:1. Reduce Motion must disable spring animations.

#04 Fix MarkdownParseCache Data Race
~1 hour Quick Win CRITICAL
MarkdownParseCache uses @unchecked Sendable — concurrent streaming updates from multiple bots could corrupt the dictionary. Fix: convert to actor or use OSAllocatedUnfairLock for thread-safe access. This is a correctness issue, not performance.
Pros
  • Eliminates potential crash during multi-bot streaming
  • 1 hour max effort — tiny change, massive safety improvement
  • actor or lock both work; actor is more Swifty
Cons
  • actor adds async overhead (negligible for cache lookups)
  • May need to audit other @unchecked Sendable uses
✅ DO — This session. 1 hour to prevent a class of rare but catastrophic crashes. Non-negotiable.

#11 Message Encryption at Rest
~4 hours v1.0 HIGH
Messages are stored as plain JSON in MessageStore. Anyone with device access can read them. Fix: AES-256-GCM via CryptoKit (allowed per GUARDRAILS — Foundation framework). Encryption key stored in Keychain alongside pairing tokens.
Pros
  • CryptoKit is zero-dependency (Foundation)
  • Protects sensitive bot conversations at rest
  • Aligns with CP02 (User Privacy First) guardrail
  • Builds user trust for App Store launch
Cons
  • Adds read/write latency (~1-2ms per message, negligible)
  • Key rotation strategy needed for multi-device
  • Encrypted data can't be searched without decrypting
  • Migration: must encrypt existing unencrypted messages
✅ DO — v1.0. Simple implementation with CryptoKit. Encrypt on write, decrypt on read. Search can operate on decrypted data in memory. Essential for privacy-conscious users.
