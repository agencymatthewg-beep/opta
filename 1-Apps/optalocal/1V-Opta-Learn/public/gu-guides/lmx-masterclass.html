MLX Native Inference
Inference
LMX uses Apple's MLX framework to run quantized models directly on the unified memory of M-series chips. The Mac Studio M3 Ultra's 192GB of unified memory (shared by CPU, GPU, and ANE) lets LMX load models that would require a data-center GPU elsewhere. Primary backend — used unless the model is GGUF-only. 
POST /v1/chat/completions
# Works identically to OpenAI — any openai SDK points here
{
  "model": "mlx-community/Mistral-7B-Instruct-v0.3-4bit",
  "messages": [{"role":"user","content":"Hello"}],
  "stream": true
}
Key Details
Backend moduleinference/mlx_lm_backend.py wraps mlx-lm for all generation calls
Batchingmodels.use_batching=true enables concurrent request batching
Context windowcontrolled by model config; engine tracks prompt token counts
Autotuneautotune_scoring.py selects optimal generation params per model

GGUF Fallback Backend
Inference
For models distributed only as GGUF (llama.cpp format), LMX routes inference through llama-cpp-python. The gguf_resolver.py detects format automatically — callers use the same API regardless of backend. Useful for community quantized models not yet in MLX format. 
# gguf_resolver.py detects format from model path
POST /admin/models/load
{
  "model_id": "TheBloke/Mistral-7B-GGUF",
  "backend_hint": "gguf"   # optional — auto-detected otherwise
}
Key Details
Auto-detectionFile extension .gguf triggers the llama-cpp-python backend path
Same APIInference routes are identical — callers don't know which backend runs
GPU layersllama-cpp-python uses Metal GPU layers on Apple Silicon automatically
Fallback chainMLX fails → GGUF tried → HTTP 503 if both fail (never crash)

SSE Streaming
Inference
Real-time token delivery over Server-Sent Events. When stream=true, LMX yields each token chunk as a data: SSE event in exactly the OpenAI streaming format. The WebSocket endpoint (/ws/stream) offers an alternative for clients that prefer bidirectional framing. 
# SSE response format (identical to OpenAI)
data: {"choices":[{"delta":{"role":"assistant","content":"Hello"}}]}
data: {"choices":[{"delta":{"content":" there"}}]}
data: [DONE]

# WebSocket alternative
ws://192.168.188.11:1234/ws/stream
Key Details
FastAPI async genstreaming.py uses AsyncIterator — FastAPI converts to SSE automatically
HeartbeatSSE heartbeat every 30s (configurable) to keep connections alive through proxies
Admin events/admin/events is a separate SSE stream for server-side telemetry
WebSocketEnabled by default; disable via server.websocket_enabled=false

Tool / Function Calling
Inference
LMX passes tool definitions and tool_calls through the inference pipeline without parsing or validating the schema — that is the client's job. The tool_parser.py module handles the model's raw output and reconstructs OpenAI-compatible tool_calls blocks for the response. 
POST /v1/chat/completions
{
  "model": "mlx-community/Mistral-7B",
  "messages": [...],
  "tools": [{"type":"function","function":{"name":"read_file",...}}],
  "tool_choice": "auto"
}
# LMX passes tools through unchanged; parses response tool_calls
Key Details
Pass-throughTool definitions forwarded unchanged to model; LMX does not validate schemas
tool_parser.pyParses model's JSON output into structured tool_calls response blocks
Parallel callsMultiple tool_calls in one response supported (OpenAI parallel calling spec)
Model supportRequires a model trained for function calling (Mistral, Llama, Qwen, etc.)

Structured Output
Inference
structured.py enforces Pydantic schema constraints on model outputs. When response_format is set to json_schema, LMX grammar-samples the model to produce valid JSON matching the provided schema. Useful for agents that need reliable structured data from local models. 
POST /v1/chat/completions
{
  "model": "mlx-community/Qwen2.5-7B",
  "messages": [...],
  "response_format": {
    "type": "json_schema",
    "json_schema": {"name":"TaskResult","schema":{...}}
  }
}
Key Details
Grammar samplingMLX constrained decoding produces schema-valid JSON tokens only
Pydantic v2Schemas are validated server-side before sampling begins
No retriesGrammar sampling eliminates parse errors — output is always valid
Use caseAgent pipelines that feed structured LMX output into downstream tools

OpenAI API Compatibility
API Surface
LMX is a drop-in replacement for LM Studio at port 1234. Any Python openai SDK client, LangChain integration, or tool that supports a custom base_url works with zero changes. The response shape, error codes, streaming format, and model listing format all match the OpenAI specification exactly. 
# Python openai SDK — works unchanged
import openai
client = openai.OpenAI(
    base_url="http://192.168.188.11:1234/v1",
    api_key="not-required-locally"
)
response = client.chat.completions.create(
    model="mlx-community/Mistral-7B-4bit",
    messages=[{"role":"user","content":"Hello"}]
)

Anthropic API Bridge
API Surface
The /v1/messages endpoint accepts Anthropic's Messages API format, allowing the Anthropic Python SDK and Claude-compatible tools to route requests through LMX to local models. LMX translates request/response shapes between Anthropic and OpenAI formats internally. 
# Anthropic SDK pointing at LMX
import anthropic
client = anthropic.Anthropic(
    base_url="http://192.168.188.11:1234",
    api_key="not-required"
)
message = client.messages.create(
    model="local/mistral-7b",
    max_tokens=1024,
    messages=[{"role":"user","content":"Hello"}]
)
Key Details
Translation layerapi/anthropic.py converts Anthropic format to internal inference request
Streaming paritystream=true produces Anthropic SSE event types (message_delta, etc.)
System promptAnthropic system field mapped to system role message
Tool useAnthropic tool_use blocks mapped to OpenAI tool_calls internally

Embeddings
API Surface
Local vector generation via /v1/embeddings. The embedding_engine.py module runs a dedicated embedding model (separate from the chat model) to produce dense vector representations for RAG pipelines, semantic search, and clustering workflows. Fully OpenAI-compatible response format. 
POST /v1/embeddings
{
  "model": "mlx-community/all-MiniLM-L6-v2",
  "input": ["document text here", "another chunk"]
}
# Returns: {"data":[{"embedding":[0.021,...],"index":0}]}
Key Details
Separate modelEmbedding model loaded independently from chat model — both can run concurrently
Batch inputInput array processed in a single forward pass for efficiency
RAG integrationEmbeddings feed directly into the RAG store for document retrieval
DimensionsEmbedding size matches the model (MiniLM=384, nomic-embed=768, etc.)

Audio API
API Surface
LMX exposes /v1/audio/transcriptions (speech-to-text) and /v1/audio/speech (text-to-speech), matching the OpenAI audio API. Transcription uses a local Whisper model via MLX; speech synthesis uses a local TTS model. Both run fully offline on the Mac Studio without any cloud calls. 
POST /v1/audio/transcriptions
Content-Type: multipart/form-data
file=@recording.wav
model=mlx-community/whisper-large-v3

POST /v1/audio/speech
{"model":"local-tts","input":"Hello world","voice":"alloy"}
Key Details
Whisper-MLXapi/audio_transcription.py — runs whisper via MLX for Apple Silicon acceleration
TTSapi/audio_speech.py — local text-to-speech synthesis
Fully offlineNo cloud dependency — audio never leaves the Mac Studio
OpenAI formatSame multipart request/response format as OpenAI Whisper API

Model Load / Unload
Model Management
The Admin API exposes hot-swap model management. Load any Hugging Face model ID or local path into unified memory, and unload it when done to free space for another. Multiple models can be resident simultaneously (up to the 90% memory ceiling). The model manager tracks inventory and enforces LRU eviction under pressure. 
POST /admin/models/load
{"model_id": "mlx-community/Qwen2.5-Coder-7B-4bit"}

POST /admin/models/unload
{"model_id": "mlx-community/Qwen2.5-Coder-7B-4bit"}

GET /admin/models   # list all loaded models + memory usage
Key Details
Auto-downloadModel not on disk? huggingface_hub downloads it to ~/.opta-lmx/models/
Auto-loadmodels.auto_load list loads models on LMX startup
LRU evictionUnder OOM pressure, least-recently-used model unloaded automatically
ConcurrentMultiple models resident; inference semaphore serializes generation

Quantization
Model Management
LMX can quantize models to lower bit-widths (3-bit, 4-bit, 8-bit) using the MLX quantization pipeline. The child_quantize_worker.py runs quantization in a supervised subprocess to isolate memory usage. The quantized model is saved locally and registered in the inventory for future use. 
POST /admin/models/quantize
{
  "model_id": "meta-llama/Llama-3.1-8B",
  "bits": 4,
  "group_size": 64,
  "output_id": "local/llama3.1-8b-4bit"
}
# Runs in isolated subprocess; SSE progress events streamed
Key Details
3/4/8-bitLower bits = smaller VRAM, higher speed, slight quality trade-off
Supervised subprocesschild_quantize_supervisor.py monitors and kills runaway quantize jobs
group_sizeSmaller group_size = better quality at same bit depth (tradeoff: size)
Progress eventsQuantization progress streamed via /admin/events SSE for dashboard

Memory Safety
Model Management
The memory monitor enforces a hard ceiling at 90% of available unified memory. Before loading any model, psutil.virtual_memory() is checked. If loading would exceed the ceiling, the least-recently-used model is evicted first. LMX never crashes on OOM — it degrades gracefully with a 503 response and a structured log event. 
# memory.py: checked before every model load
if used_pct > 0.90:
    await evict_least_recently_used()
    if still_over_limit:
        raise HTTPException(503, "Insufficient memory")
# Never: raise SystemExit / let process crash
Key Details
90% ceilingHard limit; configurable in config.yaml as models.memory_ceiling_pct
psutilUnified memory (CPU+GPU+ANE all shared) measured via psutil.virtual_memory()
LRU evictionEvicts model with oldest last-used timestamp, not the largest model
Never crashOOM is caught at the load gate — inference itself is never interrupted

Model Presets
Model Management
Presets are named configurations that bundle a model ID, generation parameters (temperature, top_p, max_tokens), and routing hints. The preset manager loads them from config at startup, and callers reference a preset name instead of spelling out all parameters per request. 
# ~/.opta-lmx/config.yaml
presets:
  code:
    model_id: mlx-community/Qwen2.5-Coder-7B-4bit
    temperature: 0.1
    max_tokens: 4096
  chat:
    model_id: mlx-community/Mistral-7B-4bit
    temperature: 0.7
    max_tokens: 2048
Key Details
Named profilesPreset name used in model field: "model":"preset:code"
Routing aliasesconfig.routing_aliases maps task categories to model IDs
Auto-loadPresets with auto_load:true are loaded into memory on LMX start
API overridePer-request params (temperature, max_tokens) override preset defaults

RAG Pipeline
Advanced
Built-in Retrieval-Augmented Generation pipeline. Documents are chunked by chunker.py, embedded into a local vector store, and retrieved via a hybrid BM25 + dense embedding search. The reranker.py module scores retrieved chunks by relevance before injecting them into the prompt context. 
POST /admin/rag/ingest
{"path": "/Users/Shared/docs/project.md"}

POST /v1/chat/completions
{
  "model": "mlx/mistral",
  "messages": [...],
  "rag": {"enabled": true, "top_k": 5}
}
Key Details
Hybrid searchBM25 keyword + dense vector similarity, combined score for ranking
Rerankerrag/reranker.py applies a cross-encoder to re-score top BM25+embedding candidates
Chunkingrag/chunker.py splits by sentence boundaries respecting token limits
Local storagerag/store.py persists embeddings in a local SQLite-backed vector store

Skills & MCP Bridge
Advanced
LMX has a skill execution system independent of the CLI. Skills are named callable units (builtin or remote) registered in a manifest. The MCP bridge connects LMX to external MCP servers over stdio or HTTP, exposing their tools as LMX skills. A policy engine controls which skills are available per request. 
GET /admin/skills   # list registered skills
POST /admin/skills/invoke
{"skill": "read_file", "params": {"path": "/tmp/data.json"}}

# MCP bridge connects to external MCP servers
# skills/mcp_bridge.py handles stdio/HTTP MCP protocol
Key Details
skills/builtins.pySelected builtin skills auto-registered at startup (file ops, search, etc.)
MCP bridgeskills/mcp_bridge.py proxies to any MCP server, exposing tools as LMX skills
Policy gateskills/policy.py controls skill access based on auth and request context
Queued dispatchQueuedSkillDispatcher serializes concurrent skill invocations safely

Agent Graph
Advanced
LMX includes a server-side multi-step agent execution engine. The AgentGraph defines a DAG of steps; the scheduler dispatches them asynchronously; the state store persists intermediate results. OpenTelemetry tracing hooks into each step for observability. Agents run fully on-server without a client keeping a connection alive. 
POST /agents/run
{
  "graph": {
    "steps": [
      {"id":"search","skill":"web_search","params":{...}},
      {"id":"summarize","model":"preset:chat","depends_on":["search"]}
    ]
  }
}
GET /agents/{run_id}/status
Key Details
DAG executionagents/graph.py builds dependency graph; scheduler runs independent steps in parallel
State persistenceagents/state_store.py checkpoints step outputs for resume after restart
OTel tracingagents/tracing.py instruments each step with span attributes and timing
Fire-and-forgetClient submits a graph and polls status — no long-lived connection required

mDNS Discovery
Operations
LMX advertises itself on the local network via mDNS (_opta-lmx._tcp.local.) so that the CLI, Opta Code Desktop, and the LMX Dashboard can find it automatically without any manual IP configuration. The CLI's opta doctor and the dashboard's ConnectAutoSetup use mDNS browsing to discover the server. 
# config.yaml
discovery:
  mdns_enabled: true
  mdns_service_name: "_opta-lmx._tcp.local"

# CLI uses mDNS to resolve Mono512 automatically
opta doctor   # shows discovered LMX hosts on LAN
Key Details
discovery.pyUses zeroconf library to register and broadcast _opta-lmx._tcp.local service
Zero configClients on the same LAN find LMX without knowing its IP address
Mono512 onlyLMX runs on Mac Studio — MacBook never runs opta-lmx locally
Disable flagdiscovery.mdns_enabled=false for network environments that block multicast

Health & Metrics
Operations
LMX provides structured observability through multiple channels: /healthz for liveness probes, /admin/events for a real-time SSE event stream, a MetricsCollector with Prometheus-compatible counters, and OpenTelemetry spans on agent and inference operations. The runtime journal persists events to disk for post-hoc analysis. 
GET /healthz   # 200 with model + memory stats
GET /admin/events   # SSE stream of server events
GET /admin/metrics  # Prometheus-style counters
GET /admin/diagnostics  # hardware probe + config dump
Key Details
/healthzReturns loaded models, memory used%, uptime, version — used by opta doctor
EventBusmonitoring/events.py publishes ServerEvent objects; SSE endpoint subscribes
Journalmonitoring/journal.py writes structured JSONL events for long-term audit
OTelOpenTelemetryMiddleware + OpenTelemetryTracer instrument every request
