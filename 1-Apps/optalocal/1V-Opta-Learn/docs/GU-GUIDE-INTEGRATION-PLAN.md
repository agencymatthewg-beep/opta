# /gu Guide Integration Plan — Opta Learn

> Implementation plan for integrating Claude Code's `/gu` HTML workflow into `learn.optalocal.com`.
> Status: **Planning** | Owner: Matthew Byrden | Created: 2026-03-06

---

## 1. Background and Motivation

### Current System (TS Guide Objects)

Guides are TypeScript objects implementing `Guide` (slug, sections[], visual HTML strings). Rendered
by `GuideViewer.tsx` using a shared layout with sticky TOC sidebar. Generated by Gemini 3.1 with
the frontend design skill via the prompts in `prompts/gemini/`.

**Strengths:** Structured, searchable, CI-validated, cross-linked, registered in a manifest.

**Limitations:** Visual richness is constrained by the `section.visual` string. Interactive
elements (explorers, sidebars, click-to-expand feature maps) require significant per-guide React
code and cannot be prototyped quickly.

### The /gu Workflow

The `/gu` Claude Code skill generates self-contained HTML files using Opta's obsidian glass design
language. These files can include:

- Full custom CSS with `color-mix()`, `backdrop-filter`, CSS animations
- Pre-rendered interactive explorers (sidebar + detail-pane, feature grids, architecture diagrams)
- JavaScript DOM manipulation (classList toggle, scroll spy) with zero external dependencies
- Terminal-art ASCII mockups, connection diagrams, live event stream visualizations

**Observation from `gu-opta-code-masterclass.html`:** The output quality and visual density of a
single `/gu` session significantly exceeds what the current `visual-interactive-journey` template
can produce inside `GuideViewer.tsx`. The interactive feature explorer (16 features, sidebar nav,
search filter, related chips, keyboard navigation) took one session and has no equivalent in the
current Learn system.

### Decision

Adopt `/gu` as the **primary authoring workflow for L4/L5 masterclass guides**. The TS guide
system continues to handle L1-L3 guides, SEO, search indexing, and the manifest registry.
A new `format: 'gu'` discriminant on the `Guide` type routes rendering to a dedicated viewer.

## 2. Architecture

### 2.1 Two-Track Guide System

```
L1  setting-configuration       → TS guide object → GuideViewer.tsx
L2  process-workflow             → TS guide object → GuideViewer.tsx
L3  feature-deep-dive            → TS guide object → GuideViewer.tsx
L3+ visual-interactive-journey  → TS guide object → GuideViewer.tsx (visual-first mode)
L4  holistic-whole-app           → TS guide object OR /gu HTML → GUGuideViewer.tsx
L5  gu-masterclass               → /gu HTML (required) → GUGuideViewer.tsx
```

L4 existing guides remain as TS objects unless manually upgraded. All new L4/L5 guides use `/gu`.

### 2.2 HTML File Storage

Generated HTML files live under `public/gu-guides/`:

```
1V-Opta-Learn/
  public/
    gu-guides/
      opta-code-masterclass.html
      lmx-masterclass.html
      cli-masterclass.html
```

Next.js serves these as static assets at `/gu-guides/[slug].html`.

### 2.3 Guide Registry Entry (format: 'gu')

A minimal TS stub is registered in `content/guides/` for every `/gu` guide:

```typescript
// content/guides/opta-code-masterclass.ts
import type { RegisteredGuide } from './index';

export const codeDesktopGUMasterclass: RegisteredGuide = {
  slug: 'opta-code-masterclass',
  title: 'Opta Code Desktop Masterclass',
  app: 'general',
  category: 'feature',
  template: 'gu-masterclass',              // new template ID
  format: 'gu',                             // new field — triggers GUGuideViewer
  guFile: 'gu-guides/opta-code-masterclass.html',
  summary: 'Complete visual guide to Opta Code Desktop — TUI layout, daemon connection, '
         + 'LMX routing, agentic loop, and 16 interactive features.',
  tags: ['code-desktop', 'tui', 'daemon', 'lmx', 'agent', 'browser', 'vault'],
  sections: [],   // empty for /gu guides — all content is in the HTML file
  updatedAt: '2026-03-06',
  status: 'verified',
};
```

### 2.4 GUGuideViewer Component

A new `components/GUGuideViewer.tsx` renders the guide in an iframe within the Learn shell.
The HTML file is fully self-contained and does not need the parent shell's CSS — the iframe
provides clean isolation with no CSS conflicts:

```
┌─ Learn Nav (fixed, same as all guides) ───────────────────────────────────┐
│  · optalocal / learn / [app] · [title]                     [search]       │
├───────────────────────────────────────────────────────────────────────────┤
│  ← Back to guides    [title]    [app chip]    [updated date]              │
├───────────────────────────────────────────────────────────────────────────┤
│                                                                            │
│   <iframe                                                                 │
│     src="/gu-guides/[slug].html"                                          │
│     title="[guide title]"                                                 │
│     style="width:100%; height:calc(100vh - 112px); border:none"          │
│   />                                                                      │
│                                                                            │
└────────────────────────────────────────────────────────────────────────── ┘
```

The iframe approach was chosen because:
- `/gu` HTML files are fully self-contained (CSS + JS + content in one file)
- No CSS collision with Tailwind in the parent page
- The guide's own scroll context is correct for its sticky sidebars and TOC
- The guide footer links (`help.optalocal.com`, `learn.optalocal.com`) work naturally

### 2.5 Routing

Extend `app/guides/[slug]/page.tsx` to branch on format:

```typescript
import { GUGuideViewer } from '@/components/GUGuideViewer';
import { GuideViewer } from '@/components/GuideViewer';

export default function GuidePage({ params }: { params: { slug: string } }) {
  const guide = getGuideBySlug(params.slug);
  if (!guide) notFound();

  if (guide.format === 'gu' && guide.guFile) {
    return <GUGuideViewer guide={guide} />;
  }
  return <GuideViewer guide={guide} />;
}
```

## 3. Type System Changes

Three additions to `content/guides/index.ts`:

```typescript
// 1. New template ID
export type GuideTemplateId =
  | 'holistic-whole-app'
  | 'feature-deep-dive'
  | 'process-workflow'
  | 'setting-configuration'
  | 'visual-interactive-journey'
  | 'gu-masterclass';                  // NEW

// 2. New fields on Guide interface
export interface Guide {
  slug: string;
  title: string;
  app: AppSlug;
  category: Category;
  template: GuideTemplateId;
  summary: string;
  tags: string[];
  sections: GuideSection[];
  updatedAt: string;
  format?: 'ts' | 'gu';               // NEW — omit or 'ts' = existing behaviour
  guFile?: string;                     // NEW — path under public/ (gu guides only)
}
```

The `format` field is optional and defaults to `'ts'` for all existing guides.
No existing guide objects need to change.

## 4. Validation Changes

Update `scripts/validate-guides.mjs` to add a `/gu` branch:

```javascript
// For guides where guide.format === 'gu':
//   REQUIRE: guFile is set
//   REQUIRE: public/${guFile} exists on disk at build time
//   REQUIRE: title, summary, tags are non-empty
//   REQUIRE: sections is an empty array (not undefined, not populated)
//   REQUIRE: updatedAt matches YYYY-MM-DD
//   SKIP:    section count, word count, visual count, lifecycle stage checks
//            (all content is in the HTML file, not in sections[])
```

This keeps CI green without applying section/word thresholds that don't apply to HTML guides.

```
Step 1: GENERATE
  cd 1V-Opta-Learn/

  → Claude Code creates ~/Synced/Opta/gu-[topic].html

  mv ~/Synced/Opta/gu-[topic].html \
     public/gu-guides/[slug].html

Step 3: REGISTER
  Create:  content/guides/[slug].ts  (use stub template from §2.3)
  Update:  content/guides/index.ts   (import + add to allGuides as status: 'draft')

Step 4: VERIFY LOCALLY
  npm run guides:validate
  npm run dev
  Visit /guides/[slug] — confirm Learn nav + iframe renders correctly

Step 5: MARK VERIFIED
  Set status: 'verified' in index.ts
  Commit and push
```

## 6. Quality Standard for /gu Guides

All `/gu` guides submitted for `status: 'verified'` must meet:

| Criterion | Requirement |
|-----------|-------------|
| Interactive explorer | Present for L5 masterclasses (16+ features, 5+ categories) |
| Feature coverage | All primary app surfaces represented |
| Visual density | At least one CSS-art mockup per major section |
| Architecture diagram | Required for any app with daemon / network components |
| Tips per feature panel | 4 tips minimum |
| Related navigation | 2+ related chips per explorer panel |
| Search filter | Working search in explorer sidebar |
| Keyboard navigation | Arrow key navigation through explorer items |
| Mobile breakpoint | `@media (max-width: 700px)` CSS present |
| Opta aesthetic | `#09090b` background, `#8b5cf6` accent, glass panels |
| No external dependencies | No CDN scripts, no external font loads, no remote images |
| JS safety | All DOM manipulation via classList, textContent, style — no eval |

## 9. Open Questions

**Q1: Stignore / sync scope**
`public/gu-guides/*.html` files can be 50–100 KB each. Confirm they should sync via Syncthing
(recommended — they are authored content, not build artifacts) rather than being gitignored and
regenerated at deploy time.

**Q2: Search depth**
Should full-text search index the HTML content of `/gu` guides? This requires an HTML parser at
build time. Defer to Phase 4 — title/tags/summary coverage is sufficient for guide discovery.

**Q3: Dedicated /gulearn skill variant**
The current `/gu` skill writes to `~/Synced/Opta/gu-*.html`. A Learn-specific variant
(`/gulearn`) could write directly to `public/gu-guides/` and also auto-create the TS stub.
Evaluate after Phase 2 to see if the manual promotion step is friction worth eliminating.

**Q4: /gu guide footer links**
The footer in `/gu` HTML files currently links to `help.optalocal.com` and `learn.optalocal.com`.
When embedded in an iframe on `learn.optalocal.com`, the footer links open externally. Consider
whether this is correct behaviour or if the footer should be suppressed in embedded mode (via a
`?embedded=1` query param checked in the HTML).
