Opta LMX Masterclass

The complete, holistic guide to the Local Model eXecution engine.

Opta LMX is the high-performance local inference engine at the heart of the stack. While you might first install the platform using Opta Init, LMX is the actual background daemon that runs on your machine permanently.

It acts as the invisible bridge. If you are typing commands into the Opta CLI, viewing active models via the Opta Local Dashboard, or even syncing preferences with Opta Accounts, they all communicate with the underlying LMX engine.

Architecture & Memory

LMX solves the hardest problem in local AI: Out-Of-Memory (OOM) crashes. It does this via a three-tier memory abstraction that makes local models feel as reliable as cloud APIs.

Tier 1: VRAM

The fastest tier. LMX attempts to load 100% of model layers into your dedicated GPU or Apple Silicon unified memory first.

Tier 2: Sys RAM

If VRAM is full, LMX dynamically spills remaining layers to system RAM. Inference slows slightly, but the app never crashes.

Tier 3: Disk Swap

During massive context operations (100k+ tokens), LMX actively swaps older KV cache context states to your NVMe storage.

Feature Deep-Dive

Zero-Downtime Hot-Swaps

LMX stays running even when no models are loaded. You can switch models instantly via the CLI. LMX gracefully dumps the VRAM and loads the new GGUF file from disk without restarting the port listener.

Air-Gapped Telemetry

LMX generates extensive performance logs, but this data is stored strictly locally. If you wish to back up your config across machines, you must explicitly link them using Opta Accounts.

Integrated Workflows

Because LMX is unopinionated about how you connect to it, it serves as the ultimate backend for the entire ecosystem and your custom tools.

The "Opta Standard" Flow

The recommended way to interact with LMX locally is combining it with the rest of the Opta stack:

  1. Install the system using Opta Init. This sets up the daemon to run on boot.
  2. Use the Opta CLI in your terminal for fast, developer-centric interactions (e.g. `opta run phi-3`).
  3. For visual management, open the Opta Local Dashboard to see VRAM usage, downloaded models, and active port connections.
