SCP Upload + Packaged CLI Fallback Flow
✓ Confirmed
Three new functions power the fallback update path. runScpUpload uploads any file to the remote host with BatchMode=yes, ConnectTimeout=15, and ServerAliveInterval=10 flags to prevent hangs. createLocalCliPackArtifact invokes npm pack --silent to produce a portable tarball. installLocalCliPackOnRemote uploads it to /tmp/, installs globally via npm install -g, then cleans up. The fallback triggers automatically when verifyRemoteCliCommands detects missing command entries and a localAppsRoot is available.
update.ts:562-593 update.ts:595-638 update.ts:645-695 update.ts:1590-1692
Build Script with Install Fallback + Forced Link
✓ Confirmed
CLI_UPDATE_BUILD_SCRIPT is a compound shell one-liner that attempts typecheck && build directly, and if that fails (missing deps), re-runs npm install --no-fund --no-audit first, then retries. After a successful build, it runs npm link --force || npm link so the binary is always globalised even when a stale link exists from a prior install.
# update.ts:99 CLI_UPDATE_BUILD_SCRIPT = ((npm run -s typecheck && npm run -s build) || (npm install --no-fund --no-audit && npm run -s typecheck && npm run -s build)) && (npm link --force || npm link)

Windows Guards on Remote + Local Update
✓ Confirmed
Two platform guards prevent crash-prone execution on Windows. The remote SSH update path exits early with a clear message ("Remote SSH update is not supported on Windows") because native OpenSSH + SCP interaction is unreliable. The local update path (POSIX build shell) similarly exits because it relies on && / || chaining that Cmd and PowerShell don't support.

Why It Changed
root cause → remediation
BEFORE — Remote update was fragile
Updates to Mono512 required the remote side to already have a working Opta CLI with all deps installed. If node_modules was stale or npm link was broken, the remote CLI would silently fail to update.
AFTER — Packaged CLI fallback
If remote verify fails, the CLI packs itself locally → SCPs it → installs it from tarball. This is a self-bootstrapping update path that works even on a clean remote machine.
BEFORE — Build could hang on missing deps
npm run build would fail cryptically if node_modules hadn't been installed first (e.g. after a fresh clone or failed prior install).
AFTER — Install-on-failure retry
The compound build script automatically reinstalls deps on first failure then retries. npm link --force handles the case where a stale global link points to a different version.
BEFORE — Linux remotes not considered
Root candidate discovery only knew about macOS /Users/ paths. Any Debian/Ubuntu remote would fail to find the CLI source directory.
AFTER — Platform-aware discovery
Linux /home/${user}/ paths are now included in the candidate list. dedupePaths prevents duplicates when user names overlap with path segments.
BEFORE — Windows would crash or hang
The remote SSH path used POSIX shell features; the local update path assumed a Unix-compatible shell. On Windows, both silently failed or produced misleading errors.
AFTER — Early platform guard with message
Platform guards now exit immediately on Windows with actionable messages. This sets up a clear extension point for a future native Windows update path.
BEFORE — health.ts used bash -lc on PATH lookup
Using bash -lc "which opta" loads the user's login profile, which is unpredictable and unavailable on Windows entirely.
AFTER — Native binary resolution per platform
where (Windows) / which (POSIX) are called directly. No bash dependency. Predictable in all CI and headless environments.
BEFORE — doctor --fix forced localhost
Autofix always wrote localhost to connection.host, wiping out valid Mono512 LAN IPs that users had configured for local AI inference.
AFTER — Prefer existing configured fallback
The fix now checks fallbackHosts first. A known good host is preserved. localhost is only used as a last resort when no fallbacks exist at all.

Residual Gaps
4 gaps · NOT yet fixed
G1
SSH Defaults Are Machine-Specific (Mono512-hardcoded)
In src/core/config.ts, the Zod schema defaults for SSH configuration point to absolute paths on Matthew's Mono512 Mac Studio. Any other user running opta update --remote without first configuring these values will silently target non-existent paths and fail with confusing errors.
config.ts:133 — user: 'opta' config.ts:134 — lmxPath: '/Users/Shared/312/...' config.ts:135 — pythonPath: '/Users/opta/.mlx-env/bin/python'
G2
opta key show Does Not Reflect Keychain/Cloud Key Source
The KeySource type in src/commands/key.ts only knows about 'env', 'config', and 'unset'. The OAuth flow in http-server.ts can sync keys from Supabase cloud into the config, and a future keychain integration would store keys in the OS keychain. Neither is surfaced in opta key show, so users cannot tell where their active key came from.
key.ts:11 — KeySource type key.ts:83 — resolveConfiguredInferenceKey http-server.ts:174+ — OAuth key sync
G3
No Per-Host Inference API Keys (apiKeysByHost)
connection.adminKeysByHost already exists as a z.record() map for per-host admin keys, proving the pattern works. But inference API keys have only one slot (connection.apiKey). Users who run multiple LMX instances on different hosts (e.g. Mono512 + a remote cloud box) must configure a single key and cannot have per-host inference keys.
config.ts:129 — adminKeysByHost (template) config.ts:128 — apiKey (only single slot)
G4
health.ts Only Checks Local Daemon — No Remote LMX Health
runHealth runs two checks in parallel: CLI surface and local daemon runtime (127.0.0.1:9999/v3/health). It does not check whether the LMX inference server at connection.host:1234 is alive. A user with a broken Mono512 connection sees a green health report from the local daemon, then hits confusing inference errors only at chat time.
health.ts:141 — runHealth (only 2 checks) health.ts:90 — checkDaemonRuntime

Add checkLmxRuntime(config: OptaConfig): Promise<HealthResult> to health.ts. Read connection.host + connection.port from config. If host is 'localhost' or '127.0.0.1' only, prompt user to configure a remote host for full health check.
02.
Use the existing checkDaemonRuntime timeout pattern (AbortSignal.timeout). Return a HealthResult discriminated union: { ok: true, latencyMs } or { ok: false, reason: 'timeout' | 'refused' | 'auth-error' | 'unexpected' }.

05.

Expand KeySource to Include Keychain + Cloud
key.ts — widen KeySource type + update resolver
High · Do Third
The OAuth key sync in http-server.ts writes keys to connection.apiKey in config. A future keychain integration would store to the OS keychain. Neither path is reflected in opta key show. The fix has two parts: (1) widen the KeySource union and (2) update resolveConfiguredInferenceKey to detect the provenance of a key when it exists.
01.
Extend KeySource at key.ts:11: 'env' | 'config' | 'keychain' | 'cloud-synced' | 'unset'. The 'cloud-synced' value means the key was written by the OAuth callback flow (can detect via a metadata flag or by checking connection.apiKeySource if added to config).
02.
Add an optional connection.apiKeySource field to the Zod schema (z.enum(['manual', 'cloud-synced', 'keychain']).optional()). The OAuth callback in http-server.ts should write this alongside the key itself.
03.
Update resolveConfiguredInferenceKey to read apiKeySource and return the correct KeySource discriminant. If the key exists and apiKeySource === 'cloud-synced', return 'cloud-synced'.
04.
Update opta key show output to display the source label with context: "Key: sk-ant-… (source: cloud-synced via OAuth)" vs "Key: sk-ant-… (source: config file)".
⚠ Risk & Rollback
Low risk — additive change only. The new apiKeySource config field is optional so existing configs continue to work. The KeySource union expansion is backward-compatible for any switch statements as long as exhaustive checks use a default case. Rollback: remove apiKeySource from config schema, revert KeySource to 3-variant union.
