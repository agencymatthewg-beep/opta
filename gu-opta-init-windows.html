What's broken / missing
0% Rollout

Opta Code Desktop

The opta-code-universal component has "percentage": 0 and empty "artifacts": { "windows": [] } in the stable manifest. Manager can't install it.

0% Rollout

Opta Daemon

The opta-daemon stable manifest has "percentage": 0. The Windows binary exists at GitHub (opta-daemon-windows-x64.zip in the beta channel) but isn't published to stable.

Arch Limit

Opta LMX (Inference)

LMX uses Apple MLX — Apple Silicon only, architecturally unavailable on Windows. The manager correctly shows 0% rollout. Windows users connect to a remote Mac Studio (Mono512) over LAN or future cloud inference.

Untested

SetupWizard on Windows

The multi-step onboarding wizard (SetupWizard.tsx) invokes Rust commands for path detection, Node.js checks, and daemon discovery. Windows paths (AppData vs ~/.config) need validation.

Untested

DaemonDrawer Process Management

The daemon status/kill/restart flow uses Rust tokio::process. Process management on Windows (signals, PID tracking, service-style restart) behaves differently than macOS.

Untested

TunnelWizard

TunnelWizard.tsx — unclear if this flow has been exercised on Windows. Needs a Windows test run to surface any path or networking edge cases.

UX Warning

Windows SmartScreen

Unsigned binary triggers "Windows protected your PC" on every fresh install. Users can bypass via More info → Run anyway. Not a functional blocker — just friction. Deferred to public release.

Missing

LMX Connection Flow (Windows)

Windows users can't run LMX locally. The manager's LMX connection UI needs to surface "connect to remote host" as the primary path on Windows, not "install LMX". Currently unclear from the UI.

Next
Validate SetupWizard Windows paths
Walk through the Rust side of SetupWizard. Config paths on Windows are %APPDATA%\opta not ~/.config/opta. The dirs crate handles this correctly in theory — verify that the wizard detects Node.js, the CLI, and daemon using Windows-correct paths.
6
Next
Validate DaemonDrawer process management on Windows
The daemon start/stop/restart flow sends OS signals on macOS (SIGTERM, SIGKILL). On Windows, process termination uses TerminateProcess via Tokio. Test that kill, restart, and status polling all work correctly on your PC.
7
Later
Windows Firewall rule for mDNS
mDNS uses UDP port 5353. Windows Firewall may block this by default, preventing the manager from discovering Opta LMX on LAN. The NSIS installer can add a firewall rule during install — or the manager can prompt the user when discovery fails.
8
Later
Node.js detection and install guidance
The CLI is an npm package — Windows users need Node.js. The SetupWizard should detect Node, show its version, and guide users to install it via winget (winget install OpenJS.NodeJS) if missing. macOS uses Homebrew — the wizard likely assumes brew.
R
Public Release Only
Windows Authenticode code signing
Eliminates the SmartScreen "Windows protected your PC" warning. Requires purchasing an EV or OV code signing certificate (~$300–500 AUD/year). Until then, SmartScreen users can bypass via More info → Run anyway. This is a UX issue, not a functional blocker — deferred until you decide to go public.
R
Public Release Only
Microsoft Store / MSIX packaging
MSIX is the modern Windows packaging format — cleaner installs, automatic updates via the Store, no SmartScreen on Store downloads. Tauri supports MSIX bundling. Optional, but worth considering as an alternative to Authenticode for store distribution.
Deferred to public release — you don't need these to be functional
💡 The rule: signing and paid platform features only get raised when you say public release is the target, OR when a missing signature actually blocks the app from launching or installing (not just a UX warning).
