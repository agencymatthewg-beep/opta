What is OpenClaw?

Core Concept

Remote Device
Command Relay

OpenClaw is the system that lets any authorized client (mobile app, web dashboard, external service) reach inside your local Opta daemon and execute operations — model loads, vault syncs, chat sessions, LMX mutations — via a cloud relay at accounts.optalocal.com, with no port forwarding required.

Your daemon polls the cloud for pending commands, executes them locally against LMX at 192.168.188.11:1234, and posts results back. The cloud relay never touches your inference data — it only carries the command envelope and result.

Layer 1 — Scope Identity

Every OpenClaw command carries a scope seed — a user-provided string or session ID. The daemon derives a deterministic 24-hex agent ID from it using SHA-256, which gets injected into every LMX inference call as the X-Client-ID and X-OpenClaw-Agent-ID headers.

// Priority: scope → actor → bridgeSessionId resolveOpenClawScopeSeed({ scope, actor, bridgeSessionId }) → "my-user-session-42" deriveOpenClawAgentId("my-user-session-42") → "opta-bridge-a3f9c2e8d1b04..." SHA-256[:24] prefix

This means: LMX can attribute inference usage to specific external callers. Quota tracking, audit logs, per-user rate limits — all possible without storing tokens.

Layer 2 — Bridge State FSM

The bridge lifecycle is a finite state machine with 5 states. Only transitions, never out-of-band mutations. The public activationState maps bridge status to the broader daemon activation model.

Backoff: 1s → 2s → 4s → 8s (max). 3 consecutive failures → degraded. Any 401/403 → unauthorized immediately.

Layer 3 — BridgeOutboundWorker (bridge-worker.ts)

The worker runs inside the daemon. It polls GET /api/device-commands/stream?deviceId=<uuid> every 300ms when idle, processes each command in sequence, and POSTs results. Two execution paths:

// Path A — LMX proxy mutation // command.payload = { method, path, body } if parseLmxProxyMutationPayload(command.payload): applyOpenClawIdentityToMutation(payload, scopeSeed) → adds X-Client-ID, X-OpenClaw-Agent-ID, body.user → POST to LMX directly // Path B — Daemon operation // command.command = "vault.pull", "models.load", etc else: executeDaemonOperation({ id, input, confirmDangerous }) → runs local structured operation

Identity injection only fires on these inference paths (not admin/health/metrics):

/v1/chat/completions /v1/completions /v1/responses

The models.skills operation gets special scope forwarding via applyOperationScope() — if the command includes no scope but a bridge scope is known, it's injected into the payload automatically.

Critical: If caller already set X-Client-ID, the worker does NOT override it. Only gap-fills.

Tier 1 — Critical: Server Control Plane

T1 These are the server-side routes the bridge worker calls. Without them, OpenClaw is client-only with no cloud relay. The bridge worker polls these endpoints every 300ms.
☁️
1R-Opta-Accounts
Server Control Plane API

The cloud relay backend. Auth-gated Next.js API routes that accept commands from web/mobile clients, store them in Supabase, stream them to the device daemon, and receive results back.

What it does: POST /api/device-commands — Authenticated web/mobile client enqueues a command. Verifies device trust_state (not revoked/quarantined), checks user owns the device, writes to Supabase, returns command ID. Rate limited: 120 req/min.

GET /api/device-commands/stream — The daemon polls this. Returns pending commands for a deviceId. 204 = nothing pending, command array = execute these.

POST /api/device-commands/{id}/result — Daemon posts execution result: { status: 'completed'|'failed'|'denied', result, error }
OpenClaw role This IS the OpenClaw relay. Layer 4 in the architecture. The bridge-worker.ts in the daemon talks exclusively to these endpoints. Every remote model load, vault sync, or LMX mutation flows through these routes.

1R-Opta-Accounts
Device Pairing System

The mobile-to-desktop pairing ceremony. A web/mobile client creates a pairing session QR code, the desktop app scans it, claims the session, and a bridge token is minted for that device pair.

What it does: POST /api/pairing/sessions — Create a pairing session (deviceId, deviceLabel, capabilityScopes, ttlSeconds). Returns a session with a scannable token/QR code.

GET /api/pairing/sessions/{id} — Poll status (pending → claimed → expired).

POST /api/pairing/sessions/{id}/claim — Desktop daemon claims the session, receives bridge token credentials.
OpenClaw role This is how a device gets its bridgeToken — the credential the BridgeOutboundWorker uses to authenticate every poll to /stream and every result POST. Without pairing, the bridge worker has no token → stuck in degraded state immediately.
src/app/api/pairing/sessions/ src/app/api/pairing/sessions/[id]/claim/

1R-Opta-Accounts
Bridge Token Mint

Mints short-lived bridge tokens scoped to specific capability grants (e.g. device.commands.issue, device.mutation.execute). Separate from Supabase session tokens — these are device-specific credentials.

What it does: POST /api/bridge/tokens — Input: deviceId, scopes[], ttlSeconds. Validates device ownership, generates a signed token, stores token metadata in Supabase. Returns the token and its expiry. The token goes into bridge-state.ts as bridgeToken.
OpenClaw role The bridge token is the Bearer credential in every request the BridgeOutboundWorker makes: the Authorization header on stream polls, the result POSTs, and the X-Bridge-Token header sent from the LMX Dashboard to the control plane when enqueueing commands.
src/app/api/bridge/tokens/route.ts

1R-Opta-Accounts
CLI Token Exchange

The CLI auth handoff routes. After a browser-based OAuth login, the CLI needs to exchange its temporary code for a real session token — these routes implement that exchange securely.

What it does: POST /api/cli/exchange — Accepts a one-time code from the CLI's localhost callback listener. Validates the code against a pending session, exchanges it for a Supabase session, returns access_token + refresh_token to the CLI process. Codes are single-use with 5min TTL.
OpenClaw role Indirectly: the CLI's access_token is what vault.pull uses to call /api/sync/keys. When the user runs opta vault pull after signing in, the CLI has a valid token from this exchange to authenticate the sync. The token also seeds the bridge pairing flow.

Pairing & Console Pages

7 new dashboard pages (arena, quantize, setup, pair, console, predictor, presets) that expose LMX capabilities through the web UI — including the pairing flow to register desktop devices.

What each does: pair/ — QR-code pairing UI using createPairingSession() + claimPairingSession(). The user flow: web shows QR → daemon CLI scans → token minted → bridge connected.

arena/ — Model comparison UI. Sends competing chat.completions commands to different loaded models via the bridge.

quantize/ — Triggers MLX quantization jobs on Mono512 remotely.

predictor/ — Performance prediction for models vs hardware specs.

presets/ — Named parameter bundles (model + generation config) for one-click loads.
OpenClaw role The pair/ and console/ pages ARE the OpenClaw web UI (Layer 5). They use accounts-control-plane.ts — which calls the Tier 1 server routes — to remotely control the desktop daemon from the browser. The rest use it indirectly via device commands.

Two new GitHub Actions workflows: one that gates PRs against the control plane contract, and one that monitors web performance budgets across all Vercel-deployed sites.

What each does: opta-accounts-control-plane-pr.yml — On every PR touching 1R routes: runs control plane lifecycle tests, validates schema migrations are present, checks bridge token mint/revocation flows. Blocks merge if control plane breaks.

OpenClaw role opta-accounts-control-plane-pr.yml directly protects OpenClaw — it gates every change to the server relay (Tier 1) behind the lifecycle test suite. Prevents silent breakage of the bridge.
